## 実装について
### パスワード・認証（※用語解説参照）情報の管理
認証機能を実装する上での大きな壁は、パスワードや認証情報の保存・管理が必要になることだと認識しています。<br>
DB管理にする場合、どんな形で管理するか・誰が保守するのかが問題になります。<br>
また、パスワードはハッシュ化した上でソルト処理を行うことが一般的ですが、そのソルトをどこでどのように管理して、誰が保守するのかも課題になります。<br>

それらを全て、Cognitoが担ってくれます。<br>

### Token
CognitoはデフォルトでIDトークン・アクセストークン・リフレッシュトークンの3種類を発行してくれます。<br>
用途ごとに使い分けることができるため、実装時に迷うことがありませんでした。<br>

| Token種別 | 用途 | 送信先 | デフォルト有効期限 |
|------|------|------|------|
| IDトークン | ユーザーの属性情報（名前・メールなど）を含む・画面表示に使用 | フロントエンドのみ | 1時間 |
| アクセストークン | 認証済みであることを証明 | バックエンド | 1時間 |
| リフレッシュトークン | IDトークン・アクセストークンを再発行 | Cognito | 30日 |

※有効期限はCognitoの設定で変更可能です。
システムのセキュリティ要件に応じて調整してください。

### エラーレスポンス
Cognitoはエラーレスポンスにエラーコードを含めて返してくれます。<br>
エラーコードからUser Poolの状態を判断できるため、状況に応じた画面表示の出し分けが可能です。<br>

コード例
```
if (エラーコードがNotAuthorizedExceptionの場合) {
  // ユーザープールの状態：パスワードリセット済み or 無効化
  // → 強制ログアウト
} else if (エラーコードがUserNotFoundExceptionの場合) {
  // ユーザープールの状態：アカウント削除済み
  // → 強制ログアウト
} else {
  // Cognito側の障害の可能性
  // → フェイルオープンで継続（業務影響を最小化）
}
```

ログイン画面に遷移したい状態をif文で判断することが可能でした。<br>

| エラーコード | ユーザープールの状態 | 処理 |
|------|------|------|
| NotAuthorizedException | パスワードリセット済み or アカウント無効化 | 強制ログアウト |
| UserNotFoundException | アカウント削除済み | 強制ログアウト |

それ以外にも公式では多くのエラーコードが用意されています。<br>
うまく使い分けることにより、よいユーザー体験を生むことが期待できます。<br>
[https://docs.aws.amazon.com/ja_jp/cognitoidentity/latest/APIReference/CommonErrors.html:embed:cite]


## ユーザー管理画面は、AWSのGUIにて行うこともできる
ユーザー管理ツールのネックなところは、管理画面が必要なところだと認識しています。<br>

* **閲覧権限**：参照のみ許可・変更不可
* **変更権限**：参照許可・変更許可

運用や設定で、考えることは山のようです。<br>
テストケースも踏まえると頭を抱えるぐらいに考えないといけません。<br>

しかし、CognitoはAWSのマネジメントコンソール（ブラウザから操作できるAWSの管理画面）上で管理できるため、これらの課題をシンプルに解決できます。<br>
IAMポリシーを使用して設定することで、Cognitoの管理画面にアクセスできるユーザーを制御・制限することが可能です。<br>

| 用途 | AWSマネージドポリシー名 |
|------|------|
| 閲覧のみ | AmazonCognitoReadOnly |
| ユーザー管理・変更 | AmazonCognitoPowerUser |
| 全権限 | AmazonCognitoFullAccess |

また、Cognitoはセキュリティ上の仕様として、管理者であってもユーザーのパスワードを参照することはできません。<br>
パスワードの再設定はCognitoのデフォルト機能として提供されています。<br>
